   128в†’   927в†’   927в†’
   129в†’   928в†’   928в†’3.3.1 Architectural Design
   130в†’   929в†’   929в†’
   131в†’   930в†’   930в†’The Cloud-Based Blood Bank Management System is built on a three-tier
   132в†’   931в†’   931в†’client-server architecture hosted on a cloud platform. This architecture
   133в†’   932в†’   932в†’separates the system into three distinct layers, each with a defined
   134в†’   933в†’   933в†’responsibility, improving maintainability, scalability, and security.
   135в†’   934в†’   934в†’
   136в†’   935в†’   935в†’Presentation Layer (Client Tier)
   137в†’   936в†’   936в†’
   138в†’   937в†’   937в†’The presentation layer is the user-facing component of the system,
   139в†’   938в†’   938в†’delivered as a responsive web application accessible through any
   140в†’   939в†’   939в†’standard web browser on both desktop and mobile devices. The frontend is
   141в†’   940в†’   940в†’built using HTML5, CSS3, and JavaScript. Upon login, the system
   142в†’   941в†’   941в†’identifies the user's role and renders only the modules and navigation
   143в†’   942в†’   942в†’items applicable to that role, in line with the TAM-informed design
   144в†’   943в†’   943в†’principles described in Chapter Two. No software installation is
   145в†’   944в†’   944в†’required at the user's end, making the system immediately accessible to
   146в†’   945в†’   945в†’any facility with internet access.
   147в†’   946в†’   946в†’
   148в†’   947в†’   947в†’Application Layer (Business Logic Tier)
   149в†’   948в†’   948в†’
   150в†’   949в†’   949в†’The application layer is the server-side component that handles all
   151в†’   950в†’   950в†’business logic, data processing, and communication between the client
   152в†’   951в†’   951в†’and the database. It is implemented using Python with the Django web
   153в†’   952в†’   952в†’framework, exposing a RESTful API that the frontend communicates with
   154в†’   953в†’   953в†’over HTTPS. The application layer handles user authentication and
   155в†’   954в†’   954в†’session management, role-based permission enforcement, blood unit
   156в†’   955в†’   955в†’lifecycle management, expiry monitoring, internal blood request
   157в†’   956в†’   956в†’processing, and the broadcast request workflow.
   158в†’   957в†’   957в†’
   159в†’   958в†’   958в†’The broadcast request workflow is a core architectural feature of the
   160в†’   959в†’   959в†’application layer. When a blood bank officer submits a network blood
   161в†’   960в†’   960в†’request, the application server broadcasts a notification to all
   162в†’   961в†’   961в†’registered facilities. Facilities with matching available blood units
   163в†’   962в†’   962в†’receive an alert on their dashboard. Blood bank officers at responding
   164в†’   963в†’   963в†’facilities review the request and submit responses through the API. The
   165в†’   964в†’   964в†’requesting facility sees all responses in real time and selects the
   166в†’   965в†’   965в†’preferred option. Upon selection, the application server updates the
   167в†’   966в†’   966в†’status of the chosen unit to Reserved at the responding facility for the
   168в†’   967в†’   967в†’duration of the reservation period.
   169в†’   968в†’   968в†’
   170в†’   969в†’   969в†’Data Layer (Database Tier)
   171в†’   970в†’   970в†’
   172в†’   971в†’   971в†’The data layer consists of a relational database hosted on the cloud,
   173в†’   972в†’   972в†’implemented using MySQL. All system data, including facility records,
   174в†’   973в†’   973в†’user accounts, blood unit records, donor information, internal requests,
   175в†’   974в†’   974в†’network requests, responses, and audit logs, are stored in the database.
   176в†’   975в†’   975в†’The cloud-hosted database provides automated backups, point-in-time
   177в†’   976в†’   976в†’recovery, and replication for high availability, ensuring that blood
   178в†’   977в†’   977в†’inventory data is never permanently lost due to hardware failure.
   179в†’   978в†’   978в†’
   180в†’   979в†’   979в†’The three tiers communicate as follows: the user's browser sends HTTPS
   181в†’   980в†’   980в†’requests to the Django application server; the application server
   182в†’   981в†’   981в†’processes the request, applies business logic and permission checks,
   183в†’   982в†’   982в†’queries or updates the database as required, and returns a JSON
   184в†’   983в†’   983в†’response; the browser renders the response as an updated user interface.
   185в†’   984в†’   984в†’All communication between the browser and the application server is
   186в†’   985в†’   985в†’encrypted using HTTPS.
   187в†’   986в†’   986в†’
   188в†’   987в†’   987в†’3.3.2 Database Design
   189в†’   988в†’   988в†’
   190в†’   989в†’   989в†’The database schema is designed around the core entities of the blood
   191в†’   990в†’   990в†’bank system and their relationships. The nine primary entities and their
   192в†’   991в†’   991в†’key attributes are described below:
   193в†’   992в†’   992в†’
   194в†’   993в†’   993в†’Facilities
   195в†’   994в†’   994в†’
   196в†’   995в†’   995в†’Stores information about each registered hospital or clinic on the
   197в†’   996в†’   996в†’network. Key attributes include a unique facility identifier, facility
   198в†’   997в†’   997в†’name, facility type (Clinic, District Hospital, Regional Hospital,
   199в†’   998в†’   998в†’Teaching Hospital), physical location, contact information, registration
   200в†’   999в†’   999в†’date, and status (Pending, Active, Suspended).
   201в†’  1000в†’  1000в†’
   202в†’  1001в†’  1001в†’Users
   203в†’  1002в†’  1002в†’
   204в†’  1003в†’  1003в†’Stores staff account information for all registered users across all
   205в†’  1004в†’  1004в†’facilities. Key attributes include a unique user identifier, a foreign
   206в†’  1005в†’  1005в†’key referencing the facility, full name, email address, bcrypt-hashed
   207в†’  1006в†’  1006в†’password, role (System Administrator, Blood Bank Officer, Laboratory
   208в†’  1007в†’  1007в†’Technician, Hospital Administrator), and account status. The role field
   209в†’  1008в†’  1008в†’governs access permissions throughout the system.
   210в†’  1009в†’  1009в†’
   211в†’  1010в†’  1010в†’Donors
   212в†’  1011в†’  1011в†’
   213в†’  1012в†’  1012в†’Maintains basic records of blood donors registered at each facility. Key
   214в†’  1013в†’  1013в†’attributes include a unique donor identifier, a foreign key referencing
   215в†’  1014в†’  1014в†’the registering facility, donor name, contact details, date of birth,
   216в†’  1015в†’  1015в†’blood group, Rhesus factor, and date of last donation. This table
   217в†’  1016в†’  1016в†’supports traceability and enables staff to contact a donor if a
   218в†’  1017в†’  1017в†’post-donation issue is identified.
   219в†’  1018в†’  1018в†’
   220в†’  1019в†’  1019в†’Storage Units
   221в†’  1020в†’  1020в†’
   222в†’  1021в†’  1021в†’Records the physical storage equipment within each facility. Key
   223в†’  1022в†’  1022в†’attributes include a unique storage unit identifier, a foreign key
   224в†’  1023в†’  1023в†’referencing the facility, storage unit name or label, temperature
   225в†’  1024в†’  1024в†’category (Refrigerated: 1 to 6 degrees Celsius; Frozen: minus 18 degrees
   226в†’  1025в†’  1025в†’Celsius and below; Room Temperature: 20 to 24 degrees Celsius), and
   227в†’  1026в†’  1026в†’maximum capacity. This supports appropriate storage assignment for
   228в†’  1027в†’  1027в†’different blood component types.
   229в†’  1028в†’  1028в†’
   230в†’  1029в†’  1029в†’Blood Units
   231в†’  1030в†’  1030в†’
   232в†’  1031в†’  1031в†’The central inventory table of the system. Key attributes include a
   233в†’  1032в†’  1032в†’unique unit identifier, foreign keys referencing the facility, the
   234в†’  1033в†’  1033в†’donor, and the assigned storage unit, blood group, Rhesus factor,
   235в†’  1034в†’  1034в†’component type (Whole Blood, Red Blood Cells, Platelets, Fresh Frozen
   236в†’  1035в†’  1035в†’Plasma), collection date, expiry date, current status (Available,
   237в†’  1036в†’  1036в†’Reserved, Issued, Expired, Discarded), and a foreign key referencing the
   238в†’  1037в†’  1037в†’staff member who registered the unit. Status transitions are validated
   239в†’  1038в†’  1038в†’by the application layer to prevent invalid state changes.
   240в†’  1039в†’  1039в†’
   241в†’  1040в†’  1040в†’Internal Blood Requests
   242в†’  1041в†’  1041в†’
   243в†’  1042в†’  1042в†’Records blood requests submitted from clinical wards or departments
   244в†’  1043в†’  1043в†’within the same facility. Key attributes include a unique request
   245в†’  1044в†’  1044в†’identifier, foreign keys referencing the facility and the requesting
   246в†’  1045в†’  1045в†’user, blood group, Rhesus factor, quantity, urgency level (Routine,
   247в†’  1046в†’  1046в†’Urgent, Emergency), patient reference, request timestamp, status
   248в†’  1047в†’  1047в†’(Pending, Approved, Issued, Rejected, Cancelled), and a foreign key
   249в†’  1048в†’  1048в†’referencing the blood unit eventually issued.
   250в†’  1049в†’  1049в†’
   251в†’  1050в†’  1050в†’Network Blood Requests
   252в†’  1051в†’  1051в†’
   253в†’  1052в†’  1052в†’Records broadcast blood requests submitted by a facility seeking blood
   254в†’  1053в†’  1053в†’from the wider network. Key attributes include a unique request
   255в†’  1054в†’  1054в†’identifier, foreign keys referencing the requesting facility and the
   256в†’  1055в†’  1055в†’requesting user, blood group, Rhesus factor, quantity, urgency level,
   257в†’  1056в†’  1056в†’broadcast timestamp, expiry timestamp, and status (Open, Partially
   258в†’  1057в†’  1057в†’Responded, Reserved, Fulfilled, Expired, Cancelled).
   259в†’  1058в†’  1058в†’
   260в†’  1059в†’  1059в†’Network Request Responses
   261в†’  1060в†’  1060в†’
   262в†’  1061в†’  1061в†’Records each response submitted by a facility in reply to an open
   263в†’  1062в†’  1062в†’network request. Key attributes include a unique response identifier,
   264в†’  1063в†’  1063в†’foreign keys referencing the network request, the responding facility,
   265в†’  1064в†’  1064в†’and the specific blood unit being offered, response timestamp, and
   266в†’  1065в†’  1065в†’status (Pending, Selected, Rejected, Withdrawn).
   267в†’  1066в†’  1066в†’
   268в†’  1067в†’  1067в†’Audit Logs
   269в†’  1068в†’  1068в†’
   270в†’  1069в†’  1069в†’Records all significant data modification events for traceability and
   271в†’  1070в†’  1070в†’regulatory compliance. Key attributes include a unique log identifier, a
   272в†’  1071в†’  1071в†’foreign key referencing the user who performed the action, the action
   273в†’  1072в†’  1072в†’type (Create, Update, Status Change, Delete), the affected record type
   274в†’  1073в†’  1073в†’and identifier, a timestamp, and the user's IP address. Audit logs are
   275в†’  1074в†’  1074в†’write-once and cannot be modified or deleted by any user role, ensuring
   276в†’  1075в†’  1075в†’a tamper-resistant record of all system activity.
   277в†’  1076в†’  1076в†’
   278в†’  1077в†’  1077в†’The relationships between these entities are structured as follows: each
   279в†’  1078в†’  1078в†’Facility has many Users, Donors, Storage Units, Blood Units, Internal
   280в†’  1079в†’  1079в†’Requests, and may create many Network Requests and submit many Network
   281в†’  1080в†’  1080в†’Responses. Each Blood Unit belongs to one Facility and one Storage Unit,
   282в†’  1081в†’  1081в†’references one Donor, and may be linked to one Internal Request or one
   283в†’  1082в†’  1082в†’Network Response. Each Network Request belongs to one Facility and may
   284в†’  1083в†’  1083в†’have many Network Responses. These relationships are enforced through
   285в†’  1084в†’  1084в†’foreign key constraints in the database, maintaining referential
   286в†’  1085в†’  1085в†’integrity across all tables.
   287в†’  1086в†’  1086в†’
   288в†’  1087в†’  1087в†’3.3.3 User Interface Design
   289в†’  1088в†’  1088в†’
   290в†’  1089в†’  1089в†’The user interface of the Cloud-Based Blood Bank Management System is
   291в†’  1090в†’  1090в†’designed around the principles of role-based visibility, minimal
   292в†’  1091в†’  1091в†’navigation depth, and clarity under time pressure. Following the
   293в†’  1092в†’  1092в†’Technology Acceptance Model framework applied in Chapter Two, every
   294в†’  1093в†’  1093в†’interface decision was guided by the objectives of maximising perceived
   295в†’  1094в†’  1094в†’usefulness and perceived ease of use for blood bank staff working in
   296в†’  1095в†’  1095в†’busy clinical environments.
   297в†’  1096в†’  1096в†’
   298в†’  1097в†’  1097в†’Login Page
   299в†’  1098в†’  1098в†’
   300в†’  1099в†’  1099в†’The login page is the entry point to the system for all users. It
   301в†’  1100в†’  1100в†’presents a clean, minimal form with fields for email address and
   302в†’  1101в†’  1101в†’password, alongside the system name. Upon successful authentication, the
   303в†’  1102в†’  1102в†’server identifies the user's role and redirects them to the appropriate
   304в†’  1103в†’  1103в†’role-based dashboard. Failed login attempts are rate-limited to prevent
   305в†’  1104в†’  1104в†’brute-force attacks.
   306в†’  1105в†’  1105в†’
   307в†’  1106в†’  1106в†’Role-Based Dashboard
   308в†’  1107в†’  1107в†’
   309в†’  1108в†’  1108в†’Each authenticated user lands on a dashboard personalised to their role.
   310в†’  1109в†’  1109в†’The Blood Bank Officer dashboard displays the most operationally
   311в†’  1110в†’  1110в†’critical information at a glance: a summary of current blood stock by
   312в†’  1111в†’  1111в†’group and Rhesus factor with indicators for low stock levels, a list of
   313в†’  1112в†’  1112в†’units expiring within the alert threshold, pending internal blood
   314в†’  1113в†’  1113в†’requests awaiting processing, and open network requests requiring
   315в†’  1114в†’  1114в†’attention. The Hospital Administrator dashboard prioritises
   316в†’  1115в†’  1115в†’facility-level overview reports and staff management. Nurses and doctors
   317в†’  1116в†’  1116в†’see a simplified view focused on submitting and tracking their blood
   318в†’  1117в†’  1117в†’requests.
   319в†’  1118в†’  1118в†’
   320в†’  1119в†’  1119в†’Blood Inventory Page
   321в†’  1120в†’  1120в†’
   322в†’  1121в†’  1121в†’The inventory page presents a searchable and filterable list of all
   323в†’  1122в†’  1122в†’blood units registered at the facility. Staff can filter by blood group,
   324в†’  1123в†’  1123в†’Rhesus factor, component type, status, and expiry date range. Each unit
   325в†’  1124в†’  1124в†’row displays its key details and current status. Blood bank officers can
   326в†’  1125в†’  1125в†’add new units through a clearly labelled button that opens a structured
   327в†’  1126в†’  1126в†’data entry form with server-side validation to prevent incomplete
   328в†’  1127в†’  1127в†’records.
   329в†’  1128в†’  1128в†’
   330в†’  1129в†’  1129в†’Network Request Page
   331в†’  1130в†’  1130в†’
   332в†’  1131в†’  1131в†’The network request page is the interface through which a blood bank
   333в†’  1132в†’  1132в†’officer initiates a broadcast request. The form captures blood group,
   334в†’  1133в†’  1133в†’Rhesus factor, quantity, urgency level, and an optional note to
   335в†’  1134в†’  1134в†’responding facilities. Once submitted, the page transitions to a live
   336в†’  1135в†’  1135в†’responses view where incoming responses from other registered facilities
   337в†’  1136в†’  1136в†’appear as they are submitted, showing the responding facility's name,
   338в†’  1137в†’  1137в†’location, the offered unit's details, and a selection button. This
   339в†’  1138в†’  1138в†’real-time response view is the primary interface through which the
   340в†’  1139в†’  1139в†’broadcast-and-response workflow is experienced by the requesting
   341в†’  1140в†’  1140в†’facility.
   342в†’  1141в†’  1141в†’

   345в†’  1144в†’  1144в†’The reports page allows hospital administrators and blood bank officers

   347в†’  1146в†’  1146в†’reporting period and report type, and the system generates a summary
   348в†’  1147в†’  1147в†’showing stock levels, units issued, units expired or discarded, and
   349в†’  1148в†’  1148в†’network request activity. Reports can be printed or exported for
   350в†’  1149в†’  1149в†’submission to management or regulatory bodies.

   352в†’  1151в†’  1151в†’3.3.4 Security Considerations
   353в†’  1152в†’  1152в†’
   354в†’  1153в†’  1153в†’Security is a foundational concern in the design of the Cloud-Based
   355в†’  1154в†’  1154в†’Blood Bank Management System, given that the system processes sensitive
   356в†’  1155в†’  1155в†’personal data belonging to blood donors. The following security measures
   357в†’  1156в†’  1156в†’are incorporated into the system design:
   358в†’  1157в†’  1157в†’
   359в†’  1158в†’  1158в†’Authentication and Session Management
   360в†’  1159в†’  1159в†’
   361в†’  1160в†’  1160в†’All users authenticate using an email address and password. Passwords
   362в†’  1161в†’  1161в†’are hashed using bcrypt before storage in the database, ensuring that
   363в†’  1162в†’  1162в†’plaintext passwords are never stored. Upon successful login, the server
   364в†’  1163в†’  1163в†’issues a session token that the browser stores securely. Sessions expire
   365в†’  1164в†’  1164в†’after a defined period of inactivity, requiring the user to
   366в†’  1165в†’  1165в†’re-authenticate.
   367в†’  1166в†’  1166в†’
   368в†’  1167в†’  1167в†’Role-Based Access Control
   369в†’  1168в†’  1168в†’
   370в†’  1169в†’  1169в†’Every API endpoint on the application server enforces role-based
   371в†’  1170в†’  1170в†’permission checks before processing a request. A nurse or doctor can
   372в†’  1171в†’  1171в†’submit a blood request but cannot access inventory management or donor
   373в†’  1172в†’  1172в†’records. A laboratory technician can update blood unit screening results
   374в†’  1173в†’  1173в†’but cannot approve or issue a unit. A blood bank officer can manage
   375в†’  1174в†’  1174в†’inventory, process requests, and interact with the network request
   376в†’  1175в†’  1175в†’workflow. A hospital administrator can manage staff accounts and view
   377в†’  1176в†’  1176в†’reports but cannot modify blood unit records. These role boundaries are
   378в†’  1177в†’  1177в†’enforced at the server level and cannot be bypassed through client-side
   379в†’  1178в†’  1178в†’manipulation.
   380в†’  1179в†’  1179в†’
   381в†’  1180в†’  1180в†’Data Encryption in Transit
   382в†’  1181в†’  1181в†’
   383в†’  1182в†’  1182в†’All communication between the user's browser and the application server
   384в†’  1183в†’  1183в†’is encrypted using HTTPS with TLS 1.2 or higher. This protects blood
   385в†’  1184в†’  1184в†’unit data, donor personal information, and authentication credentials
   386в†’  1185в†’  1185в†’from interception during transmission. The system will not serve content
   387в†’  1186в†’  1186в†’over unencrypted HTTP connections.
   388в†’  1187в†’  1187в†’
   389в†’  1188в†’  1188в†’Input Validation and SQL Injection Prevention
   390в†’  1189в†’  1189в†’
   391в†’  1190в†’  1190в†’All user-submitted data is validated on both the client side and the
   392в†’  1191в†’  1191в†’server side before processing. The Django ORM is used exclusively for
   393в†’  1192в†’  1192в†’database queries, which parameterises all inputs and provides built-in
   394в†’  1193в†’  1193в†’protection against SQL injection attacks. Input fields are constrained
   395в†’  1194в†’  1194в†’to their expected data types and formats, with server-side rejection of
   396в†’  1195в†’  1195в†’any input that does not conform to defined validation rules.
   397в†’  1196в†’  1196в†’
   398в†’  1197в†’  1197в†’Audit Trail
   399в†’  1198в†’  1198в†’
   400в†’  1199в†’  1199в†’All significant data modifications, including the creation and status
   401в†’  1200в†’  1200в†’changes of blood units, donor records, blood requests, and network
   402в†’  1201в†’  1201в†’requests, are recorded in the Audit Logs table. Each entry captures the
   403в†’  1202в†’  1202в†’user, action, affected record, timestamp, and IP address. Audit records
   404в†’  1203в†’  1203в†’are write-once and are not modifiable or deletable by any user role,
   405в†’  1204в†’  1204в†’ensuring a tamper-resistant record of all system activity.
   406в†’  1205в†’  1205в†’
   407в†’  1206в†’  1206в†’Compliance with Ghana's Data Protection Act 2012
   408в†’  1207в†’  1207в†’
   409в†’  1208в†’  1208в†’The system is designed to comply with Ghana's Data Protection Act, 2012
   410в†’  1209в†’  1209в†’(Act 843). Donor registration forms include explicit consent
   411в†’  1210в†’  1210в†’acknowledgement for the recording and processing of personal health
   412в†’  1211в†’  1211в†’data. Personal data is collected only for the purposes of blood bank
   413в†’  1212в†’  1212в†’operations and is not shared beyond the registered healthcare facilities
   414в†’  1213в†’  1213в†’on the network without additional explicit consent. These measures
   415в†’  1214в†’  1214в†’collectively ensure that the system fulfils its legal obligations under
   416в†’  1215в†’  1215в†’Ghanaian law.

   418в†’  1217в†’  1217в†’3.4 Conclusion
   419в†’  1218в†’  1218в†’
   420в†’  1219в†’  1219в†’This chapter presented the complete system specification and design of
   421в†’  1220в†’  1220в†’the Cloud-Based Blood Bank Management System. The Agile methodology was
   422в†’  1221в†’  1221в†’adopted for development, structured across four iterative sprints to
   423в†’  1222в†’  1222в†’allow the system scope to evolve in response to ongoing research
   424в†’  1223в†’  1223в†’findings and supervisor guidance. The requirements analysis, grounded in
   425в†’  1224в†’  1224в†’the project's primary survey research and field interview, produced
   426в†’  1225в†’  1225в†’twenty functional requirements and ten non-functional requirements
   427в†’  1226в†’  1226в†’covering inventory management, expiry monitoring, internal and
   428в†’  1227в†’  1227в†’cross-facility blood requests, role-based access, reporting, and
   429в†’  1228в†’  1228в†’security.
   430в†’  1229в†’  1229в†’
   431в†’  1230в†’  1230в†’The stakeholder analysis identified six distinct user roles with clearly
   432в†’  1231в†’  1231в†’defined system interactions. Ten use cases were documented to describe
   433в†’  1232в†’  1232в†’the core workflows the system is built to support, with the broadcast
   434в†’  1233в†’  1233в†’network request workflow representing the system's most significant
   435в†’  1234в†’  1234в†’functional contribution to existing blood management practice in Ghana.
   436в†’  1235в†’  1235в†’The system design presented a three-tier cloud architecture, a
   437в†’  1236в†’  1236в†’relational database schema covering nine primary entities, a
   438в†’  1237в†’  1237в†’TAM-informed user interface design built around role-based dashboards
   439в†’  1238в†’  1238в†’and minimal navigation depth, and a comprehensive security framework
   440в†’  1239в†’  1239в†’compliant with Ghana's Data Protection Act 2012.
   441в†’  1240в†’  1240в†’
   442в†’  1241в†’  1241в†’Together, these specifications and designs form the complete technical
   443в†’  1242в†’  1242в†’blueprint for the system whose implementation and testing are described
   444в†’  1243в†’  1243в†’in Chapter Four.
