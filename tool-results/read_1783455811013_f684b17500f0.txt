   926‚Üí   926‚Üí3.3 System Design
   927‚Üí   927‚Üí
   928‚Üí   928‚Üí3.3.1 Architectural Design
   929‚Üí   929‚Üí
   930‚Üí   930‚ÜíThe Cloud-Based Blood Bank Management System is built on a three-tier
   931‚Üí   931‚Üíclient-server architecture hosted on a cloud platform. This architecture
   932‚Üí   932‚Üíseparates the system into three distinct layers, each with a defined
   933‚Üí   933‚Üíresponsibility, improving maintainability, scalability, and security.
   934‚Üí   934‚Üí
   935‚Üí   935‚ÜíPresentation Layer (Client Tier)
   936‚Üí   936‚Üí
   937‚Üí   937‚ÜíThe presentation layer is the user-facing component of the system,
   938‚Üí   938‚Üídelivered as a responsive web application accessible through any
   939‚Üí   939‚Üístandard web browser on both desktop and mobile devices. The frontend is
   940‚Üí   940‚Üíbuilt using HTML5, CSS3, and JavaScript. Upon login, the system
   941‚Üí   941‚Üíidentifies the user's role and renders only the modules and navigation
   942‚Üí   942‚Üíitems applicable to that role, in line with the TAM-informed design
   943‚Üí   943‚Üíprinciples described in Chapter Two. No software installation is
   944‚Üí   944‚Üírequired at the user's end, making the system immediately accessible to
   945‚Üí   945‚Üíany facility with internet access.
   946‚Üí   946‚Üí
   947‚Üí   947‚ÜíApplication Layer (Business Logic Tier)
   948‚Üí   948‚Üí
   949‚Üí   949‚ÜíThe application layer is the server-side component that handles all
   950‚Üí   950‚Üíbusiness logic, data processing, and communication between the client
   951‚Üí   951‚Üíand the database. It is implemented using Python with the Django web
   952‚Üí   952‚Üíframework, exposing a RESTful API that the frontend communicates with
   953‚Üí   953‚Üíover HTTPS. The application layer handles user authentication and
   954‚Üí   954‚Üísession management, role-based permission enforcement, blood unit
   955‚Üí   955‚Üílifecycle management, expiry monitoring, internal blood request
   956‚Üí   956‚Üíprocessing, and the broadcast request workflow.
   957‚Üí   957‚Üí
   958‚Üí   958‚ÜíThe broadcast request workflow is a core architectural feature of the
   959‚Üí   959‚Üíapplication layer. When a blood bank officer submits a network blood
   960‚Üí   960‚Üírequest, the application server broadcasts a notification to all
   961‚Üí   961‚Üíregistered facilities. Facilities with matching available blood units
   962‚Üí   962‚Üíreceive an alert on their dashboard. Blood bank officers at responding
   963‚Üí   963‚Üífacilities review the request and submit responses through the API. The
   964‚Üí   964‚Üírequesting facility sees all responses in real time and selects the
   965‚Üí   965‚Üípreferred option. Upon selection, the application server updates the
   966‚Üí   966‚Üístatus of the chosen unit to Reserved at the responding facility for the
   967‚Üí   967‚Üíduration of the reservation period.
   968‚Üí   968‚Üí
   969‚Üí   969‚ÜíData Layer (Database Tier)
   970‚Üí   970‚Üí
   971‚Üí   971‚ÜíThe data layer consists of a relational database hosted on the cloud,
   972‚Üí   972‚Üíimplemented using MySQL. All system data, including facility records,
   973‚Üí   973‚Üíuser accounts, blood unit records, donor information, internal requests,
   974‚Üí   974‚Üínetwork requests, responses, and audit logs, are stored in the database.
   975‚Üí   975‚ÜíThe cloud-hosted database provides automated backups, point-in-time
   976‚Üí   976‚Üírecovery, and replication for high availability, ensuring that blood
   977‚Üí   977‚Üíinventory data is never permanently lost due to hardware failure.
   978‚Üí   978‚Üí
   979‚Üí   979‚ÜíThe three tiers communicate as follows: the user's browser sends HTTPS
   980‚Üí   980‚Üírequests to the Django application server; the application server
   981‚Üí   981‚Üíprocesses the request, applies business logic and permission checks,
   982‚Üí   982‚Üíqueries or updates the database as required, and returns a JSON
   983‚Üí   983‚Üíresponse; the browser renders the response as an updated user interface.
   984‚Üí   984‚ÜíAll communication between the browser and the application server is
   985‚Üí   985‚Üíencrypted using HTTPS.
   986‚Üí   986‚Üí
   987‚Üí   987‚Üí3.3.2 Database Design
   988‚Üí   988‚Üí
   989‚Üí   989‚ÜíThe database schema is designed around the core entities of the blood
   990‚Üí   990‚Üíbank system and their relationships. The nine primary entities and their
   991‚Üí   991‚Üíkey attributes are described below:
   992‚Üí   992‚Üí
   993‚Üí   993‚ÜíFacilities
   994‚Üí   994‚Üí
   995‚Üí   995‚ÜíStores information about each registered hospital or clinic on the
   996‚Üí   996‚Üínetwork. Key attributes include a unique facility identifier, facility
   997‚Üí   997‚Üíname, facility type (Clinic, District Hospital, Regional Hospital,
   998‚Üí   998‚ÜíTeaching Hospital), physical location, contact information, registration
   999‚Üí   999‚Üídate, and status (Pending, Active, Suspended).
  1000‚Üí  1000‚Üí
  1001‚Üí  1001‚ÜíUsers
  1002‚Üí  1002‚Üí
  1003‚Üí  1003‚ÜíStores staff account information for all registered users across all
  1004‚Üí  1004‚Üífacilities. Key attributes include a unique user identifier, a foreign
  1005‚Üí  1005‚Üíkey referencing the facility, full name, email address, bcrypt-hashed
  1006‚Üí  1006‚Üípassword, role (System Administrator, Blood Bank Officer, Laboratory
  1007‚Üí  1007‚ÜíTechnician, Hospital Administrator), and account status. The role field
  1008‚Üí  1008‚Üígoverns access permissions throughout the system.
  1009‚Üí  1009‚Üí
  1010‚Üí  1010‚ÜíDonors
  1011‚Üí  1011‚Üí
  1012‚Üí  1012‚ÜíMaintains basic records of blood donors registered at each facility. Key
  1013‚Üí  1013‚Üíattributes include a unique donor identifier, a foreign key referencing
  1014‚Üí  1014‚Üíthe registering facility, donor name, contact details, date of birth,
  1015‚Üí  1015‚Üíblood group, Rhesus factor, and date of last donation. This table
  1016‚Üí  1016‚Üísupports traceability and enables staff to contact a donor if a
  1017‚Üí  1017‚Üípost-donation issue is identified.
  1018‚Üí  1018‚Üí
  1019‚Üí  1019‚ÜíStorage Units
  1020‚Üí  1020‚Üí
  1021‚Üí  1021‚ÜíRecords the physical storage equipment within each facility. Key
  1022‚Üí  1022‚Üíattributes include a unique storage unit identifier, a foreign key
  1023‚Üí  1023‚Üíreferencing the facility, storage unit name or label, temperature
  1024‚Üí  1024‚Üícategory (Refrigerated: 1 to 6 degrees Celsius; Frozen: minus 18 degrees
  1025‚Üí  1025‚ÜíCelsius and below; Room Temperature: 20 to 24 degrees Celsius), and
  1026‚Üí  1026‚Üímaximum capacity. This supports appropriate storage assignment for
  1027‚Üí  1027‚Üídifferent blood component types.
  1028‚Üí  1028‚Üí
  1029‚Üí  1029‚ÜíBlood Units
  1030‚Üí  1030‚Üí
  1031‚Üí  1031‚ÜíThe central inventory table of the system. Key attributes include a
  1032‚Üí  1032‚Üíunique unit identifier, foreign keys referencing the facility, the
  1033‚Üí  1033‚Üídonor, and the assigned storage unit, blood group, Rhesus factor,
  1034‚Üí  1034‚Üícomponent type (Whole Blood, Red Blood Cells, Platelets, Fresh Frozen
  1035‚Üí  1035‚ÜíPlasma), collection date, expiry date, current status (Available,
  1036‚Üí  1036‚ÜíReserved, Issued, Expired, Discarded), and a foreign key referencing the
  1037‚Üí  1037‚Üístaff member who registered the unit. Status transitions are validated
  1038‚Üí  1038‚Üíby the application layer to prevent invalid state changes.
  1039‚Üí  1039‚Üí
  1040‚Üí  1040‚ÜíInternal Blood Requests
  1041‚Üí  1041‚Üí
  1042‚Üí  1042‚ÜíRecords blood requests submitted from clinical wards or departments
  1043‚Üí  1043‚Üíwithin the same facility. Key attributes include a unique request
  1044‚Üí  1044‚Üíidentifier, foreign keys referencing the facility and the requesting
  1045‚Üí  1045‚Üíuser, blood group, Rhesus factor, quantity, urgency level (Routine,
  1046‚Üí  1046‚ÜíUrgent, Emergency), patient reference, request timestamp, status
  1047‚Üí  1047‚Üí(Pending, Approved, Issued, Rejected, Cancelled), and a foreign key
  1048‚Üí  1048‚Üíreferencing the blood unit eventually issued.
  1049‚Üí  1049‚Üí
  1050‚Üí  1050‚ÜíNetwork Blood Requests
  1051‚Üí  1051‚Üí
  1052‚Üí  1052‚ÜíRecords broadcast blood requests submitted by a facility seeking blood
  1053‚Üí  1053‚Üífrom the wider network. Key attributes include a unique request
  1054‚Üí  1054‚Üíidentifier, foreign keys referencing the requesting facility and the
  1055‚Üí  1055‚Üírequesting user, blood group, Rhesus factor, quantity, urgency level,
  1056‚Üí  1056‚Üíbroadcast timestamp, expiry timestamp, and status (Open, Partially
  1057‚Üí  1057‚ÜíResponded, Reserved, Fulfilled, Expired, Cancelled).
  1058‚Üí  1058‚Üí
  1059‚Üí  1059‚ÜíNetwork Request Responses
  1060‚Üí  1060‚Üí
  1061‚Üí  1061‚ÜíRecords each response submitted by a facility in reply to an open
  1062‚Üí  1062‚Üínetwork request. Key attributes include a unique response identifier,
  1063‚Üí  1063‚Üíforeign keys referencing the network request, the responding facility,
  1064‚Üí  1064‚Üíand the specific blood unit being offered, response timestamp, and
  1065‚Üí  1065‚Üístatus (Pending, Selected, Rejected, Withdrawn).
  1066‚Üí  1066‚Üí
  1067‚Üí  1067‚ÜíAudit Logs
  1068‚Üí  1068‚Üí
  1069‚Üí  1069‚ÜíRecords all significant data modification events for traceability and
  1070‚Üí  1070‚Üíregulatory compliance. Key attributes include a unique log identifier, a
  1071‚Üí  1071‚Üíforeign key referencing the user who performed the action, the action
  1072‚Üí  1072‚Üítype (Create, Update, Status Change, Delete), the affected record type
  1073‚Üí  1073‚Üíand identifier, a timestamp, and the user's IP address. Audit logs are
  1074‚Üí  1074‚Üíwrite-once and cannot be modified or deleted by any user role, ensuring
  1075‚Üí  1075‚Üía tamper-resistant record of all system activity.
  1076‚Üí  1076‚Üí
  1077‚Üí  1077‚ÜíThe relationships between these entities are structured as follows: each
  1078‚Üí  1078‚ÜíFacility has many Users, Donors, Storage Units, Blood Units, Internal
  1079‚Üí  1079‚ÜíRequests, and may create many Network Requests and submit many Network
  1080‚Üí  1080‚ÜíResponses. Each Blood Unit belongs to one Facility and one Storage Unit,
  1081‚Üí  1081‚Üíreferences one Donor, and may be linked to one Internal Request or one
  1082‚Üí  1082‚ÜíNetwork Response. Each Network Request belongs to one Facility and may
  1083‚Üí  1083‚Üíhave many Network Responses. These relationships are enforced through
  1084‚Üí  1084‚Üíforeign key constraints in the database, maintaining referential
  1085‚Üí  1085‚Üíintegrity across all tables.
  1086‚Üí  1086‚Üí
  1087‚Üí  1087‚Üí3.3.3 User Interface Design
  1088‚Üí  1088‚Üí
  1089‚Üí  1089‚ÜíThe user interface of the Cloud-Based Blood Bank Management System is
  1090‚Üí  1090‚Üídesigned around the principles of role-based visibility, minimal
  1091‚Üí  1091‚Üínavigation depth, and clarity under time pressure. Following the
  1092‚Üí  1092‚ÜíTechnology Acceptance Model framework applied in Chapter Two, every
  1093‚Üí  1093‚Üíinterface decision was guided by the objectives of maximising perceived
  1094‚Üí  1094‚Üíusefulness and perceived ease of use for blood bank staff working in
  1095‚Üí  1095‚Üíbusy clinical environments.
  1096‚Üí  1096‚Üí
  1097‚Üí  1097‚ÜíLogin Page
  1098‚Üí  1098‚Üí
  1099‚Üí  1099‚ÜíThe login page is the entry point to the system for all users. It
  1100‚Üí  1100‚Üípresents a clean, minimal form with fields for email address and
  1101‚Üí  1101‚Üípassword, alongside the system name. Upon successful authentication, the
  1102‚Üí  1102‚Üíserver identifies the user's role and redirects them to the appropriate
  1103‚Üí  1103‚Üírole-based dashboard. Failed login attempts are rate-limited to prevent
  1104‚Üí  1104‚Üíbrute-force attacks.
  1105‚Üí  1105‚Üí
  1106‚Üí  1106‚ÜíRole-Based Dashboard
  1107‚Üí  1107‚Üí
  1108‚Üí  1108‚ÜíEach authenticated user lands on a dashboard personalised to their role.
  1109‚Üí  1109‚ÜíThe Blood Bank Officer dashboard displays the most operationally
  1110‚Üí  1110‚Üícritical information at a glance: a summary of current blood stock by
  1111‚Üí  1111‚Üígroup and Rhesus factor with indicators for low stock levels, a list of
  1112‚Üí  1112‚Üíunits expiring within the alert threshold, pending internal blood
  1113‚Üí  1113‚Üírequests awaiting processing, and open network requests requiring
  1114‚Üí  1114‚Üíattention. The Hospital Administrator dashboard prioritises
  1115‚Üí  1115‚Üífacility-level overview reports and staff management. Nurses and doctors
  1116‚Üí  1116‚Üísee a simplified view focused on submitting and tracking their blood
  1117‚Üí  1117‚Üírequests.
  1118‚Üí  1118‚Üí
  1119‚Üí  1119‚ÜíBlood Inventory Page
  1120‚Üí  1120‚Üí
  1121‚Üí  1121‚ÜíThe inventory page presents a searchable and filterable list of all
  1122‚Üí  1122‚Üíblood units registered at the facility. Staff can filter by blood group,
  1123‚Üí  1123‚ÜíRhesus factor, component type, status, and expiry date range. Each unit
  1124‚Üí  1124‚Üírow displays its key details and current status. Blood bank officers can
  1125‚Üí  1125‚Üíadd new units through a clearly labelled button that opens a structured
  1126‚Üí  1126‚Üídata entry form with server-side validation to prevent incomplete
  1127‚Üí  1127‚Üírecords.
  1128‚Üí  1128‚Üí
  1129‚Üí  1129‚ÜíNetwork Request Page
  1130‚Üí  1130‚Üí
  1131‚Üí  1131‚ÜíThe network request page is the interface through which a blood bank
  1132‚Üí  1132‚Üíofficer initiates a broadcast request. The form captures blood group,
  1133‚Üí  1133‚ÜíRhesus factor, quantity, urgency level, and an optional note to
  1134‚Üí  1134‚Üíresponding facilities. Once submitted, the page transitions to a live
  1135‚Üí  1135‚Üíresponses view where incoming responses from other registered facilities
  1136‚Üí  1136‚Üíappear as they are submitted, showing the responding facility's name,
  1137‚Üí  1137‚Üílocation, the offered unit's details, and a selection button. This
  1138‚Üí  1138‚Üíreal-time response view is the primary interface through which the
  1139‚Üí  1139‚Üíbroadcast-and-response workflow is experienced by the requesting
  1140‚Üí  1140‚Üífacility.
  1141‚Üí  1141‚Üí
  1142‚Üí  1142‚ÜíReports Page
  1143‚Üí  1143‚Üí
  1144‚Üí  1144‚ÜíThe reports page allows hospital administrators and blood bank officers
  1145‚Üí  1145‚Üíto generate and export inventory and usage reports. Users select a
  1146‚Üí  1146‚Üíreporting period and report type, and the system generates a summary
  1147‚Üí  1147‚Üíshowing stock levels, units issued, units expired or discarded, and
  1148‚Üí  1148‚Üínetwork request activity. Reports can be printed or exported for
  1149‚Üí  1149‚Üísubmission to management or regulatory bodies.
  1150‚Üí  1150‚Üí
  1151‚Üí  1151‚Üí3.3.4 Security Considerations
  1152‚Üí  1152‚Üí
  1153‚Üí  1153‚ÜíSecurity is a foundational concern in the design of the Cloud-Based
  1154‚Üí  1154‚ÜíBlood Bank Management System, given that the system processes sensitive
  1155‚Üí  1155‚Üípersonal data belonging to blood donors. The following security measures
  1156‚Üí  1156‚Üíare incorporated into the system design:
  1157‚Üí  1157‚Üí
  1158‚Üí  1158‚ÜíAuthentication and Session Management
  1159‚Üí  1159‚Üí
  1160‚Üí  1160‚ÜíAll users authenticate using an email address and password. Passwords
  1161‚Üí  1161‚Üíare hashed using bcrypt before storage in the database, ensuring that
  1162‚Üí  1162‚Üíplaintext passwords are never stored. Upon successful login, the server
  1163‚Üí  1163‚Üíissues a session token that the browser stores securely. Sessions expire
  1164‚Üí  1164‚Üíafter a defined period of inactivity, requiring the user to
  1165‚Üí  1165‚Üíre-authenticate.
  1166‚Üí  1166‚Üí
  1167‚Üí  1167‚ÜíRole-Based Access Control
  1168‚Üí  1168‚Üí
  1169‚Üí  1169‚ÜíEvery API endpoint on the application server enforces role-based
  1170‚Üí  1170‚Üípermission checks before processing a request. A nurse or doctor can
  1171‚Üí  1171‚Üísubmit a blood request but cannot access inventory management or donor
  1172‚Üí  1172‚Üírecords. A laboratory technician can update blood unit screening results
  1173‚Üí  1173‚Üíbut cannot approve or issue a unit. A blood bank officer can manage
  1174‚Üí  1174‚Üíinventory, process requests, and interact with the network request
  1175‚Üí  1175‚Üíworkflow. A hospital administrator can manage staff accounts and view
  1176‚Üí  1176‚Üíreports but cannot modify blood unit records. These role boundaries are
  1177‚Üí  1177‚Üíenforced at the server level and cannot be bypassed through client-side
  1178‚Üí  1178‚Üímanipulation.
  1179‚Üí  1179‚Üí
  1180‚Üí  1180‚ÜíData Encryption in Transit
  1181‚Üí  1181‚Üí
  1182‚Üí  1182‚ÜíAll communication between the user's browser and the application server
  1183‚Üí  1183‚Üíis encrypted using HTTPS with TLS 1.2 or higher. This protects blood
  1184‚Üí  1184‚Üíunit data, donor personal information, and authentication credentials
  1185‚Üí  1185‚Üífrom interception during transmission. The system will not serve content
  1186‚Üí  1186‚Üíover unencrypted HTTP connections.
  1187‚Üí  1187‚Üí
  1188‚Üí  1188‚ÜíInput Validation and SQL Injection Prevention
  1189‚Üí  1189‚Üí
  1190‚Üí  1190‚ÜíAll user-submitted data is validated on both the client side and the
  1191‚Üí  1191‚Üíserver side before processing. The Django ORM is used exclusively for
  1192‚Üí  1192‚Üídatabase queries, which parameterises all inputs and provides built-in
  1193‚Üí  1193‚Üíprotection against SQL injection attacks. Input fields are constrained
  1194‚Üí  1194‚Üíto their expected data types and formats, with server-side rejection of
  1195‚Üí  1195‚Üíany input that does not conform to defined validation rules.
  1196‚Üí  1196‚Üí
  1197‚Üí  1197‚ÜíAudit Trail
  1198‚Üí  1198‚Üí
  1199‚Üí  1199‚ÜíAll significant data modifications, including the creation and status
  1200‚Üí  1200‚Üíchanges of blood units, donor records, blood requests, and network
  1201‚Üí  1201‚Üírequests, are recorded in the Audit Logs table. Each entry captures the
  1202‚Üí  1202‚Üíuser, action, affected record, timestamp, and IP address. Audit records
  1203‚Üí  1203‚Üíare write-once and are not modifiable or deletable by any user role,
  1204‚Üí  1204‚Üíensuring a tamper-resistant record of all system activity.
  1205‚Üí  1205‚Üí
  1206‚Üí  1206‚ÜíCompliance with Ghana's Data Protection Act 2012
  1207‚Üí  1207‚Üí
  1208‚Üí  1208‚ÜíThe system is designed to comply with Ghana's Data Protection Act, 2012
  1209‚Üí  1209‚Üí(Act 843). Donor registration forms include explicit consent
  1210‚Üí  1210‚Üíacknowledgement for the recording and processing of personal health
  1211‚Üí  1211‚Üídata. Personal data is collected only for the purposes of blood bank
  1212‚Üí  1212‚Üíoperations and is not shared beyond the registered healthcare facilities
  1213‚Üí  1213‚Üíon the network without additional explicit consent. These measures
  1214‚Üí  1214‚Üícollectively ensure that the system fulfils its legal obligations under
  1215‚Üí  1215‚ÜíGhanaian law.
  1216‚Üí  1216‚Üí
  1217‚Üí  1217‚Üí3.4 Conclusion
  1218‚Üí  1218‚Üí
  1219‚Üí  1219‚ÜíThis chapter presented the complete system specification and design of
  1220‚Üí  1220‚Üíthe Cloud-Based Blood Bank Management System. The Agile methodology was
  1221‚Üí  1221‚Üíadopted for development, structured across four iterative sprints to
  1222‚Üí  1222‚Üíallow the system scope to evolve in response to ongoing research
  1223‚Üí  1223‚Üífindings and supervisor guidance. The requirements analysis, grounded in
  1224‚Üí  1224‚Üíthe project's primary survey research and field interview, produced
  1225‚Üí  1225‚Üítwenty functional requirements and ten non-functional requirements
  1226‚Üí  1226‚Üícovering inventory management, expiry monitoring, internal and
  1227‚Üí  1227‚Üícross-facility blood requests, role-based access, reporting, and
  1228‚Üí  1228‚Üísecurity.
  1229‚Üí  1229‚Üí
  1230‚Üí  1230‚ÜíThe stakeholder analysis identified six distinct user roles with clearly
  1231‚Üí  1231‚Üídefined system interactions. Ten use cases were documented to describe
  1232‚Üí  1232‚Üíthe core workflows the system is built to support, with the broadcast
  1233‚Üí  1233‚Üínetwork request workflow representing the system's most significant
  1234‚Üí  1234‚Üífunctional contribution to existing blood management practice in Ghana.
  1235‚Üí  1235‚ÜíThe system design presented a three-tier cloud architecture, a
  1236‚Üí  1236‚Üírelational database schema covering nine primary entities, a
  1237‚Üí  1237‚ÜíTAM-informed user interface design built around role-based dashboards
  1238‚Üí  1238‚Üíand minimal navigation depth, and a comprehensive security framework
  1239‚Üí  1239‚Üícompliant with Ghana's Data Protection Act 2012.
  1240‚Üí  1240‚Üí
  1241‚Üí  1241‚ÜíTogether, these specifications and designs form the complete technical
  1242‚Üí  1242‚Üíblueprint for the system whose implementation and testing are described
  1243‚Üí  1243‚Üíin Chapter Four.
